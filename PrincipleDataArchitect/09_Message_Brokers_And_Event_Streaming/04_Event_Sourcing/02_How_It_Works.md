# Event Sourcing Architecture — How It Works

## 1. The Append-Only Event Store

The core database in an Event Sourced system is called an **Event Store**. It typically looks like a chronological table with these columns:
*   `AggregateID`: The ID of the item (e.g., User ID 123).
*   `SequenceNum`: A strictly increasing integer (1, 2, 3) ensuring perfect ordering.
*   `EventType`: A string descriptor (`CartItemAdded`).
*   `Payload`: A JSON blob of data.

No row is **ever** updated or deleted. You only ever use SQL `INSERT`.

---

## 2. Rehydration (State Replay)

When a REST API wants to view User 123's Shopping Cart, the backend code queries the Event Store: `SELECT * FROM events WHERE aggregate_id = 123 ORDER BY sequence_num ASC`.

The code then passes these events through an `apply()` function:
1. `{"type": "CartCreated"}` -> Memory State: `[]`
2. `{"type": "ItemAdded", "item": "Shoes"}` -> Memory State: `["Shoes"]`
3. `{"type": "ItemAdded", "item": "Shirt"}` -> Memory State: `["Shoes", "Shirt"]`
4. `{"type": "ItemRemoved", "item": "Shoes"}` -> Memory State: `["Shirt"]`

The final returned state is `["Shirt"]`. This process is called **Rehydrating the Aggregate**.
